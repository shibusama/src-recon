# SRC 实战操作手册

## 第一层：资产战（80% 时间）

### 目标
找到别人没发现的子域名、接口、隐藏页面。

### 操作流程

拿到厂商域名后按顺序执行：

**1. 被动子域名收集**
```
你的 asset_recon.py collect_passive() 已经做了这些：
  ✅ crt.sh（证书透明度）
  ✅ HackerTarget
  ✅ RapidDNS
  ✅ AlienVault OTX
  ✅ DNSDumpster
  ✅ BufferOver.run
```

**2. DNS 爆破**（asset_recon.py 已内置 183 个常见子域名）
```
补充方向：
  - 加自动化厂商特定词库：lenovo → thinkpad、legion、yoga、moto…
  - 加业务相关词：activity、campaign、promotion、event、gift…
  - 加技术栈词：nacos、k8s、jenkins、grafana、prometheus…
```

**3. 小程序/APP 反编译抓接口**
```
微信小程序 → 反编译 wxapkg → 提取 API 地址
APP → 抓包/反编译 → 找未公开的子域名和接口
```

**4. JavaScript 扒接口**
```
目标页面 F12 → Source 找 .js 文件
asset_recon.py 可以从 JS 里提取 API 路径（如 panel.lenovo.com 的做法）
```

**5. Wayback Machine 历史存档**
```
https://web.archive.org/web/*/{domain}/*
找回旧路径、旧接口、已下线的子站
```

**6. ICP 备案查询**
```
https://beian.miit.gov.cn/
查同一备案主体下的其他域名
```

### 资产清单模板

```
目标：example.com
主站：www.example.com
子域名：（列出所有）
历史路径：（从 wayback 找到的旧接口）
小程序/APP：（反编译发现的 API）
技术栈：（Nginx/Tomcat/PHP/ASP.NET…）
优先级排序：
  P0 - 边缘子站、老旧系统、外包项目
  P1 - 新上线 72h 内的资产
  P2 - 防护明显的官网主站（最后测）
```

## 第二层：扫描策略与细节战

### 目标
同样的工具，比别人扫得更深、更准。

### 操作流程

**1. 精选字典**
```
默认字典太通用，自己再加：
  /api/、/swagger/、/docs/、/actuator/、/druid/
  /.git/、/.env/、/backup/、/test/、/debug/
  /console/、/admin/、/manager/、/upload/
```

**2. 优先级区分**
```
先扫 → 边缘资产（新发现的子域名、老旧系统）
后扫 → 官网主站（防护最严、被扫烂了）
不扫 → WAF 已经拦得很死的
```

**3. 人工逐条验证**
```
扫描器输出 → 打开浏览器逐条确认
重点关注：
  - 状态码异常（200/403/401 混合）
  - 响应长度异常的路径
  - 返回 JSON 格式的路径
```

**4. 扫描器参数调整**
```
Xray: 加 --url 指定单点深扫，不要整站泛扫
Nuclei: 先扫 critical/high，再扫 medium，最后 low
Dirsearch: 用自定义字典，加 --suffixes .bak,.old,.swp
```

## 第三层：业务逻辑漏洞（分水岭）

### 核心原理
扫描器识别不了业务逻辑漏洞，因为**所有请求看起来都是正常数据**。

### 操作清单

拿到一个需要登录的站点后，注册账号，按以下清单逐个测试：

**认证类**
```
[ ] 注册时是否可以枚举已存在的用户名/手机号/邮箱
[ ] 登录时是否存在用户名枚举（错误提示不同）
[ ] 验证码是否可复用、是否可绕过（直接删掉验证码参数）
[ ] 忘记密码的 token 是否可预测
[ ] 重置密码链接是否可被别人猜到
[ ] 手机验证码是否 6 位纯数字且无频率限制
[ ] Session 是否未在注销后失效
```

**越权类（核心）**
```
[ ] URL 中的 user_id / id / uid 改成别人的值
[ ] 订单号改成别人的订单
[ ] 接口中的参数遍历（/api/order/detail?id=1 → id=2,3,4…）
[ ] 未登录状态下直接访问需要登录的接口
[ ] 低权限用户访问高权限接口（普通用户 → admin 接口）
[ ] 改 HTTP 方法（GET 改 POST、POST 改 PUT）
[ ] 改请求头中的 role/group/admin 字段
```

**业务逻辑类**
```
[ ] 优惠券/代金券是否可以重复使用
[ ] 购买数量改为负数
[ ] 积分是否可以无限刷
[ ] 邀请奖励是否可以自己邀请自己
[ ] 支付时篡改金额
[ ] 下单后改价格再支付
[ ] 评论/评分是否可以重复提交
[ ] 投票是否可以刷票
[ ] 文件上传 ➔ 上传 jsp/php/asp webshell（改 Content-Type）
[ ] 文件上传 ➔ 上传 .htaccess / web.xml
[ ] 文件上传 ➔ 文件名注入 ../ 目录穿越
```

**短信/邮件类**
```
[ ] 短信轰炸（无时间/次数限制的重发）
[ ] 邮件轰炸
[ ] 指定接收手机号（A 用户填 B 用户的手机收验证码）
[ ] Content-Type 头导致邮件注入
```

### 测试工具
```
Burp Suite 抓包改包（足够，不需要更复杂的工具）
浏览器开发者工具（看请求/响应/本地存储）
```

## 第四层：信息差与时间差

### 目标
在新资产刚上线、别人还不知道的时候动手。

### 操作清单

**持续监控**
```
[ ] 订阅厂商的官方公众号、博客
[ ] 关注厂商的版本发布公告
[ ] 监控 GitHub 上厂商的相关仓库
[ ] 跟踪厂商的招聘信息（新业务线 = 新资产）
[ ] 关注新注册的域名（certificate transparency 实时监控）
```

**新资产窗口期**
```
活动子站、新小程序、新版 APP 刚上线 72h 内：
  - 防护还没到位
  - 扫描器还没收录
  - 漏洞密度最高
```

**小众技术栈**
```
大众内卷：ThinkPHP、WordPress、Struts2
你的机会：
  - Nacos（配置中心未授权）
  - Spring Boot Actuator（/env、/heapdump泄露）
  - 低代码平台（明道云、简道云等第三方部署）
  - 老版 OA 系统（通达、致远、泛微的历史漏洞）
  - Dubbo / ZooKeeper 未授权
```

## SRC 提交报告模板

```
漏洞标题：[厂商] [系统名] [漏洞类型]

漏洞类型：水平越权 / 逻辑漏洞 / 信息泄露 / …

影响范围：
  具体 URL / API 路径

漏洞描述：
  做了什么操作，发现了什么

复现步骤：
  1. 注册账号 A
  2. 访问 /api/order?id=1
  3. 改成 id=2 看到别人的订单
  4. 截图

影响：
  可以查看/修改其他用户的数据

修复建议：
  后端校验当前用户是否有权限访问该资源

附件：截图 / 数据包
```

## 每周固定操作流程

**周一上午（30 分钟）**
```
1. 检查关注厂商的新动态、公众号
2. 查一遍 crt.sh 看有没有新子域名
3. 更新资产清单
```

**周一~周五（每天 30-60 分钟）**
```
4. 挑 1-2 个新资产做全面收集（asset_recon.py）
5. 对边缘资产跑一遍 Nuclei critical/high
6. 找一个功能完整的站点跑业务逻辑清单
7. 整理发现的漏洞，写报告提交
```

**周末（可选）**
```
8. 复盘这周的撞洞情况
9. 更新字典和流程
```
