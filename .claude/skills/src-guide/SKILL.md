# SRC 实战四层战场操作手册

> 前置强制提醒：**所有操作仅在授权资产内执行，禁止泛扫外网；工具仅做探测，必须人工验证漏洞真实性！**

## 一、武器库：工具清单

### 1. 子域名收集（最重要，捡漏核心）

| 工具 | 类型 | 说明 |
|------|------|------|
| **crt.sh** | 网页，免装 | 查 SSL 证书历史，挖出大量被遗忘的子域名 |
| **Subfinder** | 命令行 | 白帽标配，数据源全面，速度快 |
| **OneForAll** | 命令行 | 国内开源，兼容国内 DNS 数据源，适合国内企业 |

你的 `asset_recon.py` 已经内置了 crt.sh / HackerTarget / RapidDNS / AlienVault / DNSDumpster / BufferOver 六个源，但 Subfinder 和 OneForAll 可以作为补充，覆盖更全。

### 2. 历史资产回溯

| 工具 | 说明 |
|------|------|
| **Wayback Machine** (archive.org) | 查询域名历史页面，找旧路径、旧接口、废弃后台 |
| **DomainHistory** | 配合查 DNS 历史解析记录 |

### 3. JS 源码信息提取（扫描器盲区）

| 工具 | 说明 |
|------|------|
| **JSFinder** | 爬取 JS 文件，提取隐藏 API 接口、后台路由、密钥 |

> 大量逻辑漏洞入口藏在 JS 里，扫描器不会主动解析 JS。你的 `asset_recon.py` 也可以做到类似效果（之前从 panel.lenovo.com 的 JS 里挖到了 baseURL 和 WebSocket 地址）。

### 4. 目录扫描

| 工具 | 说明 |
|------|------|
| **dirsearch** | 首选，轻量易上手 |
| **gobuster** | 速度快，适合 Linux |

> ⚠️ 重点：**不要用默认字典**，参考下文"字典优化方案"

### 5. 漏洞 POC 扫描

| 工具 | 说明 |
|------|------|
| **Nuclei** | 新手首选，大量现成模板，更新快，配置简单 |
| **Xray** | 进阶综合扫描，需要学习配置 |

### 6. 抓包改包（逻辑漏洞必备）

| 工具 | 说明 |
|------|------|
| **Burp Suite Community** | 免费版已够用，所有业务测试离不开抓包改参数 |

> 这是**区分扫描玩家和手工玩家**的核心工具。

### 7. 你的自动化资产收集脚本

```
asset_recon.py 一键完成：子域名收集 → DNS 爆破 → 端口扫描 → HTTP 探测 → 指纹识别 → 报告生成
用途：快速摸底，产出目标资产清单
```

## 二、完整资产收集操作流程（直接照着执行）

目标：某厂商主域名（如 `xxx.com`）

### 步骤 1：收集全部子域名

```
1. crt.sh 查询 %.xxx.com，导出所有子域名
2. Subfinder / OneForAll 补充扫描
3. 你的 asset_recon.py 一键跑通 6 个被动源 + DNS 爆破
4. 合并去重，剔除泛域名
```

### 步骤 2：存活探测

```
把所有子域名批量检测 HTTP/HTTPS，过滤已失效的 DNS 残留
asset_recon.py 的 http_probe() 自动做这件事
```

### 步骤 3：历史资产挖掘

```
存活资产导入 Wayback Machine，检索历史 URL
记录：
  - 旧后台路径
  - 旧上传地址
  - 曾经存在的备份文件地址
```

### 步骤 4：JS 信息挖掘

```
挑选 2-3 个边缘子站，运行 JSFinder
导出所有接口路径，加入目录扫描字典
```

### 步骤 5：分层扫描（关键，避开内卷）

```
优先级排序：
  P0 🔴 边缘资产、活动站、子域名、移动端 H5 ← 先扫这里！
  P1 🟡 老旧系统、外包项目 ← 防护弱
  P2 🔵 新上线 72h 内的资产 ← 窗口期
  P3 ⚪ 官网主站 ← 无数人 7x24 扫，最后再碰
```

## 三、扫描策略与细节战

### 字典优化方案（拉开和普通扫描仔的差距）

90% 新手只用工具自带的默认字典，高度同质化。你需要自建三类小字典合并使用：

**1. 备份文件字典**
```
bak.zip, backup.zip, www.zip, data.sql, db.sql, config.bak, web.rar
```

**2. 国内后台路径字典**
```
admin, admin.php, manage, backend, houtai, system, login
```

**3. 常见配置文件**
```
.env, config.php, application.properties, web.config
```

> 技巧：从同行业网站、历史漏洞报告里持续收集新路径，不断扩充字典。

### 扫描参数与绕过技巧

```
Xray:  加 --url 指定单点深扫，不要整站泛扫
Nuclei: 先扫 critical/high，再扫 medium，最后 low
Dirsearch: 用自定义字典，加 --suffixes .bak,.old,.swp
简单路径混淆、编码、分隔符可以绕过基础 WAF
```

### 人工复核规则

```
扫描器输出 → 打开浏览器逐条确认
重点关注：
  - 状态码异常（200/403/401 混合）
  - 响应长度异常的路径
  - 返回 JSON 格式的路径（可能是 API）
```

## 四、业务逻辑漏洞测试清单（分水岭）

### 核心原理（必读，否则你不理解为什么扫描器不行）

用大白话给你讲清楚：

**1. SQL注入、XSS 为什么扫描器能扫出来？**

```
正常访问：/info?id=1
扫描器发：/info?id=1'      ← 单引号就是攻击特征
                  ↑ 特殊符号改变了请求结构，服务器报错
```

这类漏洞**必须塞入一段异常的特殊符号才能触发**。扫描器内置一堆 Payload 挨个丢出去，看网站反应 → 自动识别。

**本质：请求本身就和正常用户访问不一样。**

**2. 越权、验证码复用、支付篡改 为什么扫描器束手无策？**

```
正常访问：/order?order_id=10086
越权改成：/order?order_id=10099
                     ↑ 只是数字变了，没有任何特殊符号！
```

数据包从头到尾和正常请求长得一模一样。扫描器不知道：
- 10086 是谁的订单
- 10099 是谁的订单
- 用户 A 能不能访问用户 B 的数据

**网站的业务逻辑只有人脑能理解，扫描器没有上帝视角。**

**3. 一句话记住**

> **有攻击特征 → 扫描器能扫（注入、XSS、文件上传）**
> **只违背业务规则 → 无统一特征 → 扫描器识别不了（逻辑漏洞）**

**4. 一个比喻**

- **扫描器 = 安检仪** — 能识别刀具、易燃易爆品（特殊 Payload）
- **业务逻辑漏洞 = 人钻规则空子** — 拿着别人的票进站，身上没有违禁品，安检仪看不出问题，**只能靠人脑发现规则缺陷**

**5. 为什么这对你很重要**

所有人都跑 Xray、dirsearch，扫到的都是**有特征的漏洞**（备份泄露、SQL 注入、XSS）。这类洞早就被轮番扫烂，重复洞泛滥。

**逻辑漏洞是扫描器盲区**，愿意沉下心手工走业务流程的人很少，竞争小、更容易中高危。

### 操作前提

拿到可登录的站点/小程序 H5，注册测试账号，**Burp 抓包逐项测试**。

### 4.1 越权漏洞（最高频，最容易中危）

```
[ ] URL 中 user_id / id / uid 改成别人的值，能否查看他人信息
[ ] 订单号改成别人的订单
[ ] 接口参数遍历（/api/order/detail?id=1 → id=2,3,4…）
[ ] 普通用户访问管理员接口
[ ] 未登录直接访问需登录的接口
[ ] 低权限用户访问高权限接口（role 参数）
[ ] 改 HTTP 方法（GET 改 POST、POST 改 PUT）
[ ] 改请求头中的 role/group/admin 字段
```

### 4.2 验证码相关漏洞

```
[ ] 验证码是否前端本地校验（F12 关掉校验直接绕过）
[ ] 验证码是否可以重复使用
[ ] 多次获取验证码，验证码不更新
[ ] 验证码是否 6 位纯数字且无频率限制（可爆破）
[ ] 重置密码 token 是否可预测
```

### 4.3 前端限制绕过

```
[ ] 购买数量、金额只在前端限制 → 抓包篡改突破
[ ] 上传文件前端校后缀 → 抓包改 Content-Type / 文件名绕过
[ ] 删除验证码参数直接放行
```

### 4.4 短信/邮件/消息相关

```
[ ] 传入多个手机号 → 是否批量发送（短信轰炸）
[ ] 是否可以篡改接收人手机号/邮箱
[ ] 无时间/次数限制的重发
[ ] Content-Type 头导致邮件注入
```

### 4.5 业务逻辑类

```
[ ] 优惠券/代金券是否可以重复使用
[ ] 购买数量改为负数
[ ] 积分是否可以无限刷
[ ] 邀请奖励是否可以自己邀请自己
[ ] 支付时篡改金额
[ ] 下单后改价格再支付
[ ] 评论/评分是否可以重复提交
[ ] 投票是否可以刷票
[ ] 文件上传 → webshell（改 Content-Type）
[ ] 文件上传 → .htaccess / web.xml 覆盖
[ ] 文件上传 → 文件名注入 ../ 目录穿越
```

### 4.6 接口访问控制

```
[ ] 退出登录后，直接复制接口 URL，能否继续访问
[ ] 不需要 Token 就能调用敏感查询接口
```

> 实操建议：**不要一上来就找高危**。先熟练挖水平越权、未授权访问，在漏洞盒子很容易评中危。

## 五、信息差与时间差（高阶赛道）

### 持续监控

```
[ ] 订阅厂商官方公众号、博客
[ ] 关注厂商版本发布公告
[ ] 监控 GitHub 上厂商的相关仓库
[ ] 关注招聘信息（新业务线 = 新资产）
[ ] 证书透明度实时监控新注册域名
```

### 新资产窗口期

活动子站、新小程序、新版 APP **刚上线 72h 内**：
- 防护还没到位
- 扫描器还没收录
- 漏洞密度最高

### 小众技术栈（避开大众内卷）

```
大众内卷：ThinkPHP、WordPress、Struts2
你的机会：
  - Nacos（配置中心未授权）
  - Spring Boot Actuator（/env、/heapdump 泄露）
  - 低代码平台（明道云、简道云等第三方部署）
  - 老版 OA 系统（通达、致远、泛微历史漏洞）
  - Dubbo / ZooKeeper 未授权
```

```
周一上午（30 分钟）
  ├─ 检查关注厂商的新动态、公众号
  ├─ crt.sh 查新子域名
  └─ 更新资产清单

周一~周五（每天 30-60 分钟）
  ├─ 挑 1-2 个新资产做全面收集（asset_recon.py）

  ├─ 找功能完整的站点跑业务逻辑清单
  └─ 整理漏洞写报告提交

周末（可选）
  ├─ 复盘本周撞洞情况

```

| 活动 | 占比 |
|------|:----:|
| 资产搜集、去重、存活检测 | **60%** |
| 定向扫描边缘资产 + 人工核验 | **25%** |
| 注册账号，手工测试业务逻辑 | **15%** |

## 七、避坑红线

```
1. ❌ 扫描结果不人工核验直接提交
   → 大量误报，账号权重降低，无法报名付费众测

2. ❌ 重复漏洞
   → 热门资产先检索厂商历史漏洞，避免白忙活

3. ❌ 长时间不间断高频扫描同一站点
   → 触发 WAF，厂商反感，审核从严

4. ❌ 任何删除数据、爆破账号、DoS 等破坏性测试

5. ❌ 认为"开更强的扫描器、多开几台机器就能卷赢"
   → 大量机器轮询主站，只产出重复漏洞，还容易封 IP

6. ✅ 正确思路
   → 自动化负责广度，人工负责深度
   → 工具用来寻找入口，人脑用来寻找漏洞
```

## 八、第一条漏洞快速产出路线（小白起步）

公益 SRC 挑选一家中小企业，按以下步骤：

```
Day 1: 资产收集
  ├─ crt.sh 查子域名（5 分钟）
  ├─ asset_recon.py --quick（2 分钟）
  └─ 找到一个冷门的二级站点

Day 1: 目录扫描
  ├─ dirsearch + 自定义备份字典
  └─ 发现备份文件 / 敏感目录 / .git 泄露

Day 1: 提交报告
  ├─ 截图 + 整理规范报告
  └─ 提交 → 首条有效漏洞 ✅
```

**常见首洞类型（按难度排序）：**

| 漏洞类型 | 难度 | 评级 |
|---------|:----:|:----:|
| 备份文件泄露 | ⭐ | 低危 |
| .git 泄露 | ⭐ | 中危 |
| Swagger UI 未授权 | ⭐ | 中危 |
| Actuator 信息泄露 | ⭐⭐ | 中危 |
| 水平越权 | ⭐⭐⭐ | 中危~高危 |
| 未授权访问 | ⭐⭐ | 中危 |

## 九、SRC 报告模板

```
漏洞标题：[厂商] [系统名] [漏洞类型]

漏洞类型：水平越权 / 逻辑漏洞 / 信息泄露 / …

影响范围：
  具体 URL / API 路径

漏洞描述：
  做了什么操作，发现了什么

复现步骤：
  1. 注册账号 A
  2. 访问 /api/order?id=1
  3. 改成 id=2 看到别人的订单
  4. 截图

影响：
  可以查看/修改其他用户的数据

修复建议：
  后端校验当前用户是否有权限访问该资源

附件：截图 / 数据包
```

## 十、命令速查模板

### Subfinder
```bash
subfinder -d example.com -silent -o subs.txt
```

### Nuclei
```bash
# 扫描 critical + high
nuclei -l alive.txt -severity critical,high -o vulns.txt

# 扫描 takeover 子域名接管
nuclei -l subs.txt -t takeovers/

# 扫描 API 安全
nuclei -t api/ -target https://api.example.com
```

### Dirsearch
```bash
dirsearch -u https://target.com -w custom_dict.txt --suffixes .bak,.old
```

### Xray
```bash
xray webscan --url https://target.com --plugins xss,sqldet,cmd-injection
```

### JSFinder
```bash
python JSFinder.py -u https://target.com -ou output.txt
```

### asset_recon.py（你的脚本）
```bash
# 快速摸底
python3 asset_recon.py --quick example.com

# 全量扫描 + nmap 端口扫描
python3 asset_recon.py example.com

# 批量扫配置中的所有域名
python3 asset_recon.py --batch
```
